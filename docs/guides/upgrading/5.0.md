# 5.0 Upgrade Guide

In this document we describe important changes to be aware of when upgrading from `orchestrator-core` v4.8 (or earlier) to v5.0.

## About 5.0

This release introduces a significant change to how workflow descriptions are managed:

1. Workflow descriptions are now managed exclusively in the database via the UI or API, rather than hardcoded in Python decorator code.
2. The `description` parameter in workflow decorators (`@create_workflow`, `@modify_workflow`, `@terminate_workflow`, `@validate_workflow`, `@reconcile_workflow`, `@workflow`) is now deprecated.
3. Validation that workflow descriptions in code match database descriptions has been removed, as descriptions can now be freely changed via the UI without updating the code.

### Why this change?

Previously, workflow descriptions had to be kept in sync in two places:
- In the Python code as a decorator parameter
- In the database as a record field

This created confusion and maintenance burden:
- Changing a description via the UI didn't match the code, causing validation failures
- Code and database descriptions could get out of sync
- Developers had to update both locations to keep things consistent

With this change, descriptions are now single-source-of-truth in the database, making the system simpler and more flexible.

### Migration steps

#### 1. Update your workflow decorators (Recommended)

Remove the `description` parameter from all workflow decorators. You'll see deprecation warnings if you don't:

**Before:**
```python
@create_workflow("Create SN8 IP BGP", initial_input_form=initial_input_form_generator)
def create_sn8_ip_bgp() -> StepList:
    # ...

@modify_workflow(
    "Change a port to a different port in a SN8 IP BGP subscription",
    initial_input_form=initial_input_form_generator,
)
def modify_sn8_ip_bgp() -> StepList:
    # ...

@terminate_workflow("Terminate SN8 IP BGP", initial_input_form=terminate_initial_input_form_generator)
def terminate_sn8_ip_bgp() -> StepList:
    # ...

@validate_workflow("Validate SN8 IP BGP")
def validate_sn8_ip_bgp() -> StepList:
    # ...
```

**After:**
```python
@create_workflow(initial_input_form=initial_input_form_generator)
def create_sn8_ip_bgp() -> StepList:
    # ...

@modify_workflow(initial_input_form=initial_input_form_generator)
def modify_sn8_ip_bgp() -> StepList:
    # ...

@terminate_workflow(initial_input_form=terminate_initial_input_form_generator)
def terminate_sn8_ip_bgp() -> StepList:
    # ...

@validate_workflow()
def validate_sn8_ip_bgp() -> StepList:
    # ...
```

If you don't remove the description parameters, you'll see deprecation warnings in the logs like:
```
Workflow decorator description is deprecated
workflow_description=Create SN8 IP BGP
hint=Remove the description parameter and manage it via database/UI instead
```

#### 2. Manage descriptions in the database

There are several ways to set workflow descriptions in the database:

##### Option A: Using the UI (Recommended for existing workflows)
1. Navigate to the Workflows section in the Orchestrator UI
2. Click on a workflow to view its details
3. Click "Edit" or the edit icon
4. Update the description field
5. Save the changes

##### Option B: Using the API
Send a PATCH request to update a workflow description:
```bash
PATCH /api/v1/workflows/{workflow_id}
Content-Type: application/json

{
  "description": "Your new description here"
}
```

##### Option C: Using migrations for new workflows
When adding new workflows to your product types using the `db migrate-workflows` CLI command, you will now be prompted to provide a description for each workflow:

```
$ python main.py db migrate-workflows "add new workflows"

Which product type should the workflow be added to?
1) my_product_type

Add product type my_product_type to which workflow?
1) create_my_product

Please provide a description for this workflow.
This description will be managed in the database and can be updated via the UI/API endpoint.
Workflow description: Create a new My Product subscription
```

#### 3. Run validation

After updating your decorators, run the validation task to ensure everything is correct:

```bash
python main.py tasks validate-products
```

This should complete without errors related to description mismatches.

### What about code generation?

If you're using the orchestrator's code generation feature to create products, generated workflow decorators will no longer include hardcoded descriptions. The descriptions will be managed entirely through database migrations and the UI.

### Backward compatibility

Workflow decorators with description parameters will continue to work during the deprecation period (5.0 - 5.x), but you'll see warnings. Future versions will remove the description parameter entirely.

### Database requirements

The `workflows.description` column in the database is `NOT NULL` and required. All workflows must have a description in the database. If you have any workflows without descriptions, you should add them before upgrading.

### Related changes

- The `check_workflows_for_matching_targets_and_descriptions` validation step no longer checks descriptions, only targets and names.
- New workflows registered via the `db migrate-workflows` CLI command will require a description to be provided by the user.
- The `workflow` decorator in `orchestrator.workflow` now also accepts an empty description (for backward compatibility during the transition period).
